1. Introduction

ShipHappens is a paper-generator for teachers. The teacher uploads 3–5 past exam papers; Gemini analyses them, extracts the structural blueprint (sections, question types, marks distribution, instruction patterns, tone), and generates a new paper of the same shape. The teacher then edits any question in the browser, regenerates individual questions on demand, and exports the result to PDF.

System Summary

A Next.js wizard walks the teacher through four steps: Upload, Blueprint Review, Editable Paper, Export. A FastAPI backend handles file uploads, Gemini calls, and persistence. Supabase provides managed Postgres (sessions, papers, source files) and Storage (uploaded PDFs). The Gemini API handles every AI call — three prompts total: blueprint extraction, paper generation, and per-question regeneration. PDFs go to Gemini directly via the Files API; there is no embedding pipeline and no vector database.

LLM Call Budget

Per session: 1 call for blueprint extraction, 1 call for paper generation, plus one fast call per question the teacher regenerates. Files are uploaded to the Gemini Files API exactly once per session and reused across all three prompts (cached as gemini_file_uri on each SourceFile row).

MVP Scope

This is a proof of concept for a 20-hour buildathon. Out of scope: authentication, multi-tenant separation, vector search / RAG, per-section regeneration, server-rendered PDF (we use the browser's print dialog), email export, paper history, and curriculum-aware difficulty calibration. Demo content is A-Level English; the architecture is subject-agnostic.

4. Data Models

Three SQLModel tables persisted in Supabase Postgres. Two JSON shapes (BlueprintSchema and PaperSchema) are the contract between the AI module and the rest of the system — both AI engineer and frontend engineer should treat these as immutable during the sprint.

4.1 SQLModel Tables (backend/db/models.py)

class Session(SQLModel, table=True):
    id: UUID = Field(default_factory=uuid4, primary_key=True)
    created_at: datetime = Field(default_factory=datetime.utcnow)
    title: str | None = None
    board: str | None = None       # e.g. "Cambridge", "AQA", "Edexcel"
    level: str | None = None       # e.g. "A Level", "AS Level"
    subject: str | None = None     # e.g. "English Literature"

class SourceFile(SQLModel, table=True):
    id: UUID = Field(default_factory=uuid4, primary_key=True)
    session_id: UUID = Field(foreign_key="session.id", index=True)
    filename: str
    storage_path: str              # uploads/{session_id}/{file_uuid}.pdf
    gemini_file_uri: str | None    # cached Gemini Files API URI
    uploaded_at: datetime = Field(default_factory=datetime.utcnow)

class Paper(SQLModel, table=True):
    id: UUID = Field(default_factory=uuid4, primary_key=True)
    session_id: UUID = Field(foreign_key="session.id", unique=True, index=True)
    blueprint_json: dict = Field(default={}, sa_column=Column(JSONB))
    paper_json: dict = Field(default={}, sa_column=Column(JSONB))
    updated_at: datetime = Field(default_factory=datetime.utcnow)

Contract Discipline

Every question.id is stable for the life of the paper. Regeneration replaces the question object's content but keeps its id. The frontend uses id as the React key. The backend uses id to locate-and-splice in regenerate.

5. API Contracts

All endpoints are application/json unless otherwise noted. :id is the session UUID; the sid cookie carries it automatically so the FE doesn't have to thread it through every call (but the path param is the source of truth).

MethodPathOwnerRequestResponse
POST /sessions FS1 { title?, board?, level?, subject? } { session_id } + Set-Cookie
POST /sessions/:id/upload FS1 + FS2 multipart/form-data with PDF files { source_files: SourceFile[] }
POST /sessions/:id/blueprint FS1 + AI — { blueprint: BlueprintSchema }
POST /sessions/:id/generate FS1 + AI — { paper: PaperSchema }
PUT /sessions/:id/paper FS1 + FS2 { paper: PaperSchema } { saved_at }
POST /sessions/:id/questions/:qid/regenerate FS1 + AI { nudge?: string } { question: Question }
GET /sessions/:id/paper/export.html FS1 + FS2 — text/html · print-ready
GET /sessions/:id FS1 — { session, blueprint?, paper? } · for resume

6. UI Component Mapping

Four wizard pages plus a shared layout. All pages route under app/(wizard)/ in the Next.js App Router and share a sticky 4-step progress indicator.

Step 1 — Upload (/upload)
  • Purpose: Collect 3–5 reference PDFs and optional metadata (title/board/level/subject)
  • Components: react-dropzone, shadcn Input for metadata, file list with remove buttons, primary Button "Analyze"
  • API calls (in order): POST /sessions, then POST /sessions/:id/upload (per file), then POST /sessions/:id/blueprint
  • State writes: sessionId, sourceFiles, blueprint
  • Next: Route to /blueprint on 200
Step 2 — Blueprint Review (/blueprint)
  • Purpose: Show what Gemini extracted so the teacher can sanity-check before generating
  • Components: shadcn Accordion (one per section), Badge for marks, read-only fields, primary Button "Generate New Paper", secondary Button "Re-analyze"
  • API calls: POST /sessions/:id/generate
  • State reads: blueprint
  • State writes: paper
  • Next: Route to /paper on 200
Step 3 — Editable Paper (/paper)
  • Purpose: Per-question editing and regeneration; this is the heart of the product
  • Components: shadcn Card per section, sub-Card per question with Textarea (auto-grow), marks Input, Button "Regenerate" (with optional Input for nudge), Skeleton during regen
  • API calls: Debounced PUT /sessions/:id/paper (800ms), POST /sessions/:id/questions/:qid/regenerate on demand
  • State reads: paper
  • State writes: paper (optimistic), lastSavedAt
  • Next: Primary Button "Preview & Export" → /export
Step 4 — Export (/export)
  • Purpose: Show the print preview and let the user save as PDF via browser print
  • Components: iframe at full width pointing to /api/proxy/export.html (Next.js API route that proxies to FastAPI), primary Button "Download PDF" calls iframe.contentWindow.print(), secondary Button "Back to Edit"
  • API calls: GET /sessions/:id/paper/export.html (via the proxy)
  • State reads: sessionId

7. Team Task Lanes

Each developer owns one vertical lane. Hour ranges are cumulative from kickoff (h0). Definition-of-done is checked at the corresponding milestone in Section 8.

AI Engineer

Owns backend/ai/ · 3 prompts · Pydantic schemas
h0–h2 SDK spike. Write a 30-line standalone script that uploads one sample PDF via genai.Client.files.upload(), calls generate_content with a Pydantic response_schema, prints the JSON. Commit to backend/ai/spike.py. Lock the SDK pattern the rest of the team will copy.

h13–h16 Per-question regeneration. Implement regenerate_question(paper, blueprint, question_id, nudge?) -> Question in backend/ai/regenerate.py. Use gemini-2.5-flash. Return only the replacement question object.
h16–h20 Prompt polish + safety. Iterate on observed output (mark consistency, instruction style, no off-topic content). Add a retry-with-parse-failure wrapper. Help debug demo-day issues.
Definition of done: Each Pydantic schema imports cleanly, each function returns a valid model from a real PDF set, retry-on-parse-failure wrapper logs raw output when it kicks in.

10.2 Scope Cuts (in priority order if we slip)

  1. Drop per-section regenerate (keep per-question only).
  2. Drop Blueprint Review step — go Upload → generated Paper directly; show the blueprint collapsed above the paper.
  3. Drop autosave on edit (require explicit "Save" button).
  4. Drop per-question regenerate (text edits only).
  5. Drop styling on PDF export (plain HTML print is acceptable for demo).
  6. Drop the optional "nudge" input on regenerate (just regenerate without it).

11. Glossary

TermDefinition
BlueprintThe structural fingerprint of a teacher's past papers — sections, marks distribution, question types, instructions style, tone. Output of the first Gemini call.
Session UUIDAnonymous identifier for one "build a paper" workflow. Stored in the sid cookie and as the primary key of the session table.
Gemini Files APIGoogle's endpoint for uploading large binary inputs (PDFs) and referencing them by URI in subsequent generate_content calls. Uploads are cached per SourceFile.
Structured OutputGemini feature where you pass a Pydantic schema as response_schema and the model is constrained to return JSON matching that schema.
Supabase StorageManaged object storage (S3-compatible) provided by Supabase. We use a single private bucket uploads.
SpliceReplacing one question object inside paper_json by question_id while preserving section + ordering.
Print CSSCSS @media print rules plus @page for A4 sizing; turns an HTML page into a paginated PDF when the browser prints to PDF.
NudgeOptional free-text instruction passed to the regenerate prompt, e.g. "harder" or "more focused on theme of guilt".
Vertical SliceAn end-to-end working path through the system at a checkpoint hour (h8, h14, h18) — used to ensure demoability is preserved over local perfection.
MVPMinimum Viable Product — this 20-hour build is scoped to four wizard steps, three Gemini calls, and one shared Supabase project.
